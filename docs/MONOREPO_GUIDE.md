# OfficePortal — Monorepo Guide

**Last updated:** 2026-05-17

Canonical reference for the monorepo layout: workspace purpose,
startup commands, Docker usage, dev workflow, CI strategy, and
environment strategy. Keep this in sync whenever workspace structure
changes.

---

## 1. Workspace purpose

```
OfficePortal/
├── backend/        FastAPI service — HRMS API, schema, business logic
├── admin-panel/    React admin console (super_admin, office_admin)
├── mobile-app/     Flutter employee app (attendance, leaves, payslips)
└── docs/           Cross-workspace design notes + this guide
```

| Workspace | Primary stack | Audience | Owner of |
|---|---|---|---|
| `backend/` | FastAPI · PostgreSQL · SQLAlchemy 2 · Alembic · Redis | Server | API endpoints, DB migrations, audit logging, RBAC, business rules |
| `admin-panel/` | Vite · React 19 · TypeScript · Tailwind v4 · TanStack Query 5 · React Router 7 | Admins | Admin UI screens, role-gated routes, admin auth flow |
| `mobile-app/` | Flutter · Dio · Riverpod · go_router · flutter_secure_storage | Employees | Mobile UI screens, employee auth flow, on-device storage |

Each workspace has its own `README.md` with stack-specific detail.

## 2. Root vs workspace ownership

Cross-cutting assets live at the **repo root**. Workspace-specific
assets live **inside the workspace**. The split:

### Root-level (monorepo concerns)

| File / dir | Purpose |
|---|---|
| `.git/` | Single repo root (git history was hoisted here in Phase 2a) |
| `.github/workflows/ci.yml` | CI workflow; path-filtered per workspace |
| `.gitignore` | Cross-cutting ignores only — OS noise, editor, env overlays, logs, coverage, `.claude/` |
| `CLAUDE.md` | Monorepo-wide rules for Claude sessions |
| `README.md` | Workspace overview + common commands |
| `docker-compose.yml` | Local dev stack (postgres + redis + migrate + app). Builds `./backend` and bind-mounts `./backend/location.json`. Orchestrates the backend, but lives at root because the admin-panel + mobile-app dev loops both depend on it. |
| `docs/` | Cross-workspace docs (this guide, PROJECT_STATUS, CURRENT_ARCHITECTURE, NEXT_STEPS) |

### Workspace-level (workspace concerns)

| Backend (`backend/`) | Admin (`admin-panel/`) | Mobile (`mobile-app/`) |
|---|---|---|
| `Dockerfile` | `package.json` / `package-lock.json` | `pubspec.yaml` |
| `.dockerignore` | `vite.config.ts` | `analysis_options.yaml` |
| `requirements.txt` / `requirements-dev.txt` | `tsconfig*.json` | `lib/` (Dart source) |
| `alembic.ini` / `pytest.ini` | `eslint.config.js` | platform dirs (`ios/`, `android/`, etc. — generated by `flutter create .`) |
| `main.py`, `app/`, `migrations/`, `tests/` | `index.html`, `public/`, `src/` | |
| `.env`, `.env.example` (Python config) | `.env.example`, `.env.local` (`VITE_*` only) | `.env.example`, `.env` (compile-time `--dart-define`) |
| `.gitignore` (Python-only) | `.gitignore` (Node-only) | `.gitignore` (Flutter/Dart-only) |
| `Readme.md` | `README.md` | `README.md` |

### `.gitignore` discipline

- **Root**: cross-cutting rules only. If an entry only applies to one
  workspace, it doesn't belong here.
- **Workspace**: only rules specific to that workspace's tooling.
  Never duplicate root rules (`.DS_Store`, `.vscode/`, `.env`, `*.log`).

Adding a new ignore rule? Decide: does this pattern appear in 2+
workspaces? Root. Only one workspace? Workspace.

## 3. Startup commands

### Fresh clone

```bash
git clone https://github.com/SarisTS/OfficePortal.git
cd OfficePortal
```

### Backend (full stack via Docker)

The fastest path. Brings up Postgres 18, Redis 7, runs migrations once,
then starts uvicorn on `:8000`.

```bash
# From repo root
cp backend/.env.example backend/.env   # fill in real values
docker compose up --build              # foreground
# OR
docker compose up --build -d           # detached
```

API: <http://localhost:8000> · Docs: <http://localhost:8000/docs> ·
Health: <http://localhost:8000/health>

Stop: `docker compose down` (keeps the db volume) or
`docker compose down -v` (drops the volume — full reset on next `up`).

### Backend (host, no Docker)

For iterating on Python without container rebuilds.

```bash
cd backend
python -m venv venv
venv\Scripts\activate                  # Windows
# source venv/bin/activate             # macOS/Linux
pip install -r requirements.txt
pip install -r requirements-dev.txt    # if running tests
cp .env.example .env                   # fill in real values
alembic upgrade head                   # against the DATABASE_URL in .env
uvicorn main:app --reload              # http://localhost:8000
```

You need a running Postgres + Redis reachable via the URLs in `.env`.
Easiest path: keep `docker compose up db redis` running while the host
uvicorn iterates.

### Admin panel

```bash
cd admin-panel
cp .env.example .env.local             # set VITE_API_URL=http://localhost:8000
npm install
npm run dev                            # http://localhost:5173
```

### Mobile app

```bash
cd mobile-app
flutter create . --project-name officeportal_mobile --org com.officeportal   # one-time
flutter pub get
cp .env.example .env                   # adjust API_URL per platform
flutter run --dart-define-from-file=.env
```

Platform notes for `API_URL`:
- Android emulator: `http://10.0.2.2:8000` (alias for host's loopback)
- iOS simulator: `http://127.0.0.1:8000`
- Physical device on same Wi-Fi: host machine's LAN IP

## 4. Docker usage

### What compose owns

Compose orchestrates the **backend stack only**:

- `db` — Postgres 18 with `pg_isready` healthcheck, volume `db_data`
  for persistence
- `redis` — Redis 7 with `ping` healthcheck
- `migrate` — one-shot `alembic upgrade head` that exits 0; `app`
  depends on it via `service_completed_successfully`
- `app` — uvicorn on `:8000`, depends on `migrate` (succeeded) +
  `redis` (healthy)

The admin-panel and mobile-app are NOT in compose. They run on the
host (`npm run dev`, `flutter run`) and connect to the API at
`http://localhost:8000`.

### Build context

`docker-compose.yml` lives at the repo root. Both `migrate` and `app`
use `build: ./backend`, so:

- `backend/Dockerfile` is the build recipe
- `backend/.dockerignore` controls what enters the image
- `env_file: ./backend/.env` injects runtime config
- `./backend/location.json:/app/location.json:ro` bind-mounts the
  ~38 MB reference dataset (gitignored, not in the image)

### Backend image alone (no compose)

```bash
cd backend
docker build -t office-portal-backend .
docker run --rm -p 8000:8000 \
  --env-file .env \
  -v "$(pwd)/location.json:/app/location.json:ro" \
  office-portal-backend
```

### Front-end and mobile in Docker

Not done today. Both run on the host. If a CI build container for
admin-panel becomes useful (e.g. for browser-based E2E in CI), add a
`Dockerfile` inside `admin-panel/` and a new compose profile rather
than expanding the root compose file.

## 5. Development workflow

### Branching

Single working branch, no per-feature branches:

```
chore/env-example  ← all work happens here
       │  (fast-forward)
       ▼
     main           ← origin/main mirrors this
```

Cycle:

1. `git checkout chore/env-example` (or `git switch -c chore/env-example` if missing)
2. Make changes; commit
3. `git checkout main && git merge --ff-only chore/env-example`
4. `git push origin main chore/env-example`

Never force-push to `main`. Never skip hooks (`--no-verify`). Never
amend already-pushed commits.

### Day-to-day

Most days you work in one workspace at a time:

- **Backend day**: `docker compose up db redis` (or just `up`) → host
  uvicorn → `pytest` for quick checks.
- **Frontend day**: `docker compose up` (full stack on `:8000`) +
  `npm run dev` (`:5173`). The Vite proxy isn't configured today —
  `VITE_API_URL` in `.env.local` points the Axios client at the
  backend directly.
- **Mobile day**: `docker compose up` + `flutter run` on an emulator
  / simulator. Pick the right `API_URL` for the device class.

### Cross-workspace changes

If a single change touches `backend/` + `admin-panel/` (e.g. a new
endpoint + the UI that consumes it), commit them together. The
commit message should make the dependency explicit.

### Adding a feature

- Backend: see `backend/Readme.md` and `backend/API_AUDIT.md`. New
  endpoints follow the `ApiResponse[T]` envelope and pass the
  `permissions.py` helper checks.
- Admin: see `docs/FRONTEND_NEXT_STEPS.md` for the per-feature
  checklist (api.ts, components/, pages/, types.ts, index.ts).
- Mobile: see `docs/MOBILE_NEXT_STEPS.md` for the per-feature
  layout.

## 6. CI strategy

CI is split per workspace via independent workflows, each path-filtered
so unrelated changes don't burn runner minutes.

### `ci.yml` — backend

Runs on push / PR to `main` when any of:

- `backend/**`
- `docker-compose.yml`
- `.github/workflows/ci.yml`

Two parallel jobs:

| Job | Purpose | Working dir |
|---|---|---|
| `test` (pytest) | Python-level test suite over SQLite in-memory + mocked Redis. 151 tests today. | `backend/` |
| `compose-smoke` (docker compose) | Builds real images, brings up Postgres + Redis + migrate + app, probes `/health` on a live container. | repo root (compose file is there) |

Concurrency group: `ci-${ref}`, cancel-in-progress.

### `admin-panel.yml` — admin panel

Runs on push / PR to `main` when any of:

- `admin-panel/**`
- `.github/workflows/admin-panel.yml`

Single job `build`:

| Step | Purpose |
|---|---|
| `npm ci` | Deterministic install (fails on lock-file drift) |
| `npm run lint` | ESLint over the whole workspace |
| `npm run build` | `tsc -b && vite build` — type-check + production bundle |
| upload-artifact `admin-panel-dist` | 7-day retention so the bundle is downloadable from a failed PR for inspection |

Concurrency group: `admin-panel-${ref}`, cancel-in-progress.

### `mobile-app.yml` — mobile app

Runs on push / PR to `main` when any of:

- `mobile-app/**`
- `.github/workflows/mobile-app.yml`

Single job `build`:

| Step | Purpose |
|---|---|
| setup-java@v4 (temurin 17) | AGP 8.x toolchain |
| subosito/flutter-action@v2 (pinned 3.27.0) | Flutter SDK + caching |
| `flutter create .` | Regenerate ios/ + android/ each run — the scaffold ships Dart-side only; this keeps the workflow valid before any contributor runs the one-time local `flutter create .`. The boilerplate `test/widget_test.dart` is deleted (references a non-existent `MyApp`). |
| `flutter pub get` | Resolve deps |
| `flutter analyze` | Lints per `analysis_options.yaml` |
| `flutter build apk --debug` | Validates code compiles, native deps resolve, Gradle accepts the project |
| upload-artifact `officeportal-mobile-debug-apk` | 7-day retention for download from a failed PR |

Concurrency group: `mobile-app-${ref}`, cancel-in-progress.

iOS build is deferred — needs macOS runners (5–10× the cost). Add a
parallel `build-ios` job on `macos-latest` when iOS release planning
kicks in. A `flutter test` step joins the workflow as soon as real
tests land under `test/`.

### Cross-workspace principle

Keep workflows independent. Don't merge them into one cross-workspace
mega-workflow — path filters do that for you, and per-workspace
failure isolation is more useful than a single matrix.

## 7. Environment strategy

Three independent `.env.example` files — one per workspace. Each is
committed; the corresponding `.env` (or `.env.local`) is gitignored.

| Workspace | Template | Live file | Loaded by |
|---|---|---|---|
| `backend/` | `.env.example` | `.env` | `pydantic-settings` at FastAPI startup; `docker-compose.yml` via `env_file:` |
| `admin-panel/` | `.env.example` | `.env.local` | Vite at build/dev time. Only `VITE_*`-prefixed vars are exposed to the client bundle. |
| `mobile-app/` | `.env.example` | `.env` | `flutter run --dart-define-from-file=.env` — values bake into the build at compile time. No runtime `.env` is shipped. |

### Why not one root `.env`

- Three different runtimes (Python, Node, Dart) with three different
  loaders.
- Different secrets exposure rules: Vite leaks anything `VITE_*`-
  prefixed into the client; Flutter bakes everything into the build
  artefact; backend `.env` is server-side only.
- Mixing them risks accidentally exposing a backend secret to a
  client bundle.

### Secrets

- `backend/.env` carries real secrets in production (DB password,
  JWT `SECRET_KEY`, Google OAuth client secret, Gmail app password).
  Rotation is the deployer's responsibility, out-of-band. Don't
  touch `.env` via Claude tooling.
- CI uses the placeholder values from `backend/.env.example` —
  `/health` and the pytest suite don't need real credentials.
- Frontend `VITE_*` vars are PUBLIC by definition (they ship in the
  bundle). Never put secrets there. Use the backend as the secret
  custody point.
- Mobile `--dart-define` values are baked into the binary and
  extractable by anyone with the APK / IPA. Treat them as public.

### Adding a new env variable

1. Add to the workspace's `.env.example` with a comment explaining
   what it does, expected format, and whether it's optional.
2. Wire it into the workspace's config layer:
   - Backend: `app/core/config.py` (pydantic `Settings`)
   - Admin: read via `import.meta.env.VITE_*` (use `@/lib/env.ts`
     for typed access)
   - Mobile: declare in `lib/core/env/env.dart` with `String.fromEnvironment(...)`
3. If it's a secret needed by CI compose-smoke, add a placeholder to
   `backend/.env.example` (CI copies the example).
4. Document it in the workspace README if non-obvious.

## 8. Cleanup / maintenance checklist

When the structure changes (file moves, new workspace, new conventions):

- [ ] Update `README.md` (root) with the new layout / commands
- [ ] Update this guide (`docs/MONOREPO_GUIDE.md`)
- [ ] Update `docs/CURRENT_ARCHITECTURE.md` if architecture shifted
- [ ] Update `docs/PROJECT_STATUS.md` "set up from scratch" block
- [ ] Update `backend/Readme.md` if backend-relative paths changed
- [ ] Adjust `.gitignore` files if any pattern moved between root
      and a workspace
- [ ] Adjust CI path filters + working-directory if files moved
- [ ] Run `docker compose config` from repo root to validate the
      compose YAML interpolates cleanly
- [ ] Run `npm run build` in `admin-panel/` to confirm the bundler
      still resolves everything
- [ ] Commit on `chore/env-example`, ff-merge to `main`, push both
